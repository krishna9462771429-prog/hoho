                      FEATURE 1: AI FAILURE DIAGNOSIS SYSTEM
================================================================================

PURPOSE:
  Intelligently analyze API failures and provide root cause analysis,
  suggested fixes, and retry recommendations.

BACKEND IMPLEMENTATION:
  File: backend/ai/failure_diagnosis.py

  Functions:
    - categorize_failure(status_code, error_message) -> str
      * Categorizes failure into: timeout, dns, rate_limit, auth,
        server_error, client_error, ssl, network, unknown

    - diagnose_failure(api_name, url, status_code, error_message,
                       latency_ms, previous_logs) -> Dict
      * Uses Groq LLaMA-3.1-8b-instant as primary AI
      * Falls back to Gemini 1.5 Flash if Groq fails
      * Structured prompt for JSON response
      * Includes context from previous 10 failures
      * Returns: diagnosis, severity, confidence, suggested_fix,
                 recommended_action, provider_used, failure_category

  File: backend/services/diagnosis_service.py

  Functions:
    - create_diagnosis(user_id, api_id, ...) -> Dict
      * Creates diagnosis record in database
      * Broadcasts diagnosis_generated event via WebSocket

    - get_diagnoses_for_api(api_id, user_id, limit) -> List[Dict]
      * Retrieves diagnosis history for specific API

    - get_diagnosis_stats(user_id) -> Dict
      * Returns: total, by_severity, by_category, by_action, avg_confidence

  API ENDPOINTS:
    POST /ai/diagnose
      Input:
        - api_name: str
        - url: str
        - status_code: int (optional)
        - error_message: str
        - latency_ms: int
        - api_id: str (optional)
        - previous_logs: List[Dict] (optional)

      Output:
        - diagnosis: str
        - severity: "low" | "medium" | "high" | "critical"
        - confidence: 0.0 - 1.0
        - suggested_fix: str
        - recommended_action: "retry" | "skip" | "escalate" |
                             "investigate" | "disable" | "contact_support"
        - provider_used: "groq" | "gemini"
        - failure_category: str
        - explanation: str (detailed analysis)

    GET /ai/diagnoses/{api_id}
      - Returns last N diagnoses for specific API

    GET /ai/diagnoses
      - Returns all diagnoses for user

    GET /ai/diagnosis-stats
      - Returns aggregate statistics

================================================================================
                      SCHEDULER JOBS IMPLEMENTATION
================================================================================

FILE: backend/scheduler/jobs.py

check_all_apis() - Every 60 seconds:
  1. Fetch all active APIs from database
  2. For each API:
     a. Make HTTP request to API endpoint
     b. Measure latency
     c. Determine success/failure based on expected_status
     d. Insert log entry into api_logs table
     e. Update API statistics (total_checks, failed_checks, uptime_percent)
     f. Broadcast api_status_update via WebSocket
     g. Broadcast new_log via WebSocket
     h. If failure detected:
        - Broadcast failure_detected
        - Create AI diagnosis via Groq/Gemini
        - Store diagnosis in failure_diagnoses table
        - Broadcast diagnosis_generated
     i. If recovery detected (was error, now success):
        - Broadcast recovery_detected

ping_all_tickers() - Every 5 minutes:
  1. Fetch all active ticker services
  2. For each ticker:
     a. Make GET request to ticker URL
     b. Measure latency
     c. Update last_pinged_at, last_status, ping_count
     d. Broadcast ticker_ping via WebSocket

TECHNOLOGIES USED:
  - APScheduler 3.10.4 (AsyncIOScheduler)
  - IntervalTrigger for periodic execution
  - httpx for async HTTP requests
  - Supabase Python SDK for database operations
  - WebSocket manager for real-time broadcasting
